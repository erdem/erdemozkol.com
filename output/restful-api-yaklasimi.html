RESTful API yaklaşımı

İlk blog yazımı okuyan sen okuyucaya merhaba! Uzun süredir sunucu (back-end) tarafında geliştirme yaparken, genelde RESTful api modeli üzerinde geliştirme yapıyorum. Bu model üzerinde geliştirme yaparken yaşadığım deneyimleri ve sorunları paylaşmak için bu yazıyı yazmaya karar verdim.

API Versiyonlama

Bir uygulama için geliştirdiğiniz bir RESTful servisinde, uygulamaya yeni özellikler eklenmesi için bazı değişiklikler gerekebilir. Bu değişiklikleri servise uyguladığınızda, uygulamanın eski versiyonunu kullanan kullanıcılar için sorun olucaktır. Uygulama daha önceki servis yapısına göre çalıştığı için, crash edebilir veya sunucudan yanlış veri alabilir. Bu gibi durumların önüne geçmek api versiyonlama yapmanız lazım.

RESTful servisleri ilk yaygınlaşmaya başladığında versiyonlama URL bazında yapılıyordu. Ancak bir süre sonra işin içinden çıkılmaz bir hal aldı. Bunu konuyu bir örnekle açıklıyayım.

url: /api/v1/books/

[{
name: "Harry Potter and the Goblet of Fire", 
author: "J._K._Rowling"
}]

url: /api/v1.1/books/

[{
name: "Harry Potter and the Goblet of Fire", 
author: "J._K._Rowling", 
release_date: "2000" 
}]

Yukardaki örnekte olduğu gibi kitapların verisini aldığımız bir api adresi olduğunu varsayalım. API v1 versiyonunda bir kitap'ın ismi ve yazarın adı verilmiş, daha sonra uygulamanın v1.1 versiyonunun da bu verilerin yanında çıkış tarihi eklenmiş. Buraya kadar herşey güzel ama bir sorun var. Geliştirdiğimiz API da artık bir kitabın birden fazla detay URL'i var.

Burada en önemli nokta servise yeni versiyon verilicek zamani belirlemek, eğer servise bağlı uygulamalar stable bir versiyona ulaştıysa, bu süreçten sonra ki geliştirmeleri yeni versiyonda yapmanız doğru bir karar olucaktır.

API versiyonlama başta çoğu servis için gerekli gözükmeyebilir, ama uzun development süreçlerinde gerçekten hayat kurtar.
